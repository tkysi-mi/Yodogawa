# ヒアリング質問と受け入れ基準の例

SKILL.md 手順2〜5でタスク定義を詰めていく際の質問・記載例。

## 目的・背景のヒアリング

- 「現状の課題・不具合は何か？」
- 「誰が困っているのか？（ユーザー/開発/運用など）」
- 「このタスクが完了すると、どのような価値・KPI 改善があるか？」

数値・頻度・Before/After を含めて具体的に記載する。

## ユーザーストーリー

### 形式

```
[役割]として、[目的]がしたい。なぜなら[理由]だから
```

### 参考質問

- 「このタスクで誰が何を達成したいですか？」
- 「その理由は？」

優先度や関連するストーリーID（US-001 など）を付与すると後続管理が容易。

## 変更内容のカテゴリ

- **画面/UI**: 新規ページ、既存フォームの項目追加
- **API/サービス**: 新規エンドポイント、仕様変更
- **データモデル/DB**: テーブル追加、カラム変更、インデックス
- **その他**: 設定、翻訳、ドキュメント更新など

可能な限り、ファイル名・エンドポイント・テーブル名を具体的に記載する。

## 受け入れ基準の観点

- **正常系**: 期待する操作が成功する
- **異常系・エラー表示**: 入力エラー、権限不足など
- **性能・セキュリティ**: 応答時間、認可/認証、暗号化
- **テスト要件**: ユニット/統合/E2E、ログ出力、監視アラート

### 具体例

- 「メール認証リンクは 1 回のみ有効」
- 「API レスポンスは 200ms 以内」
- 「CSRF トークン必須」
- 「登録完了まで 3 クリック以内」（「使いやすい」の言い換え）
