# 検証ループの詳細

SKILL.md 手順 2 の「検証ループ（最大 3 回）」で実行するフローと判断基準。

## 目的

- 実装直後に小さなフィードバックサイクルを回し、バグを次ステップへ持ち越さない。
- 「実装 → 検証 → 修正 → 再検証」を明示的な手順として回す。
- 無限ループを防ぐため最大 3 回で打ち切り、解決しなければユーザーへエスカレーションする。

## 実行フロー

```
[1] 検証コマンドを実行
      │
      ├─ 合格 ──► ステップ完了マーク へ
      │
      └─ 失敗 ──► [2] 失敗内容の分析
                        │
                        └─► [3] 修正実装
                              │
                              └─► [1] へ戻る（カウンタ +1）

        カウンタが 3 に達したら → エスカレーション
```

## 検証コマンドの構成

プロジェクトで利用可能なコマンドを優先順に実行する（未定義のコマンドはスキップ）。

| # | 種類 | 例 |
|:--|:--|:--|
| 1 | 型チェック | `npm run type-check`, `tsc --noEmit`, `mypy`, `go vet` |
| 2 | Lint | `npm run lint`, `eslint .`, `ruff check`, `golangci-lint run` |
| 3 | ユニットテスト | `npm test`, `pytest`, `go test ./...` |
| 4 | ビルド（任意） | `npm run build`, `go build ./...` |

## 失敗時の分析基準

1. **エラー種別を分類**: 型エラー / Lint 警告 / テスト失敗 / ビルドエラー
2. **エラー箇所を特定**: 行番号・関数名を抽出
3. **根本原因を推定**: 直近の変更との因果関係を確認
4. **修正範囲を最小化**: 関係ない領域へ波及させない

## エスカレーション条件

以下に該当したらユーザーに報告し、判断を仰ぐ。

- 3 回試行しても同じエラーが解消しない
- エラーが実装対象外のコードから発生している
- 仕様が曖昧でユーザーの意思決定が必要
- 外部サービス・環境要因（API 障害、依存パッケージの破壊的変更）が疑われる

## エスカレーション時の報告フォーマット

```
検証ループ 3 回実行しましたが、以下のエラーが解消しませんでした。

- 試行内容:
  1. {1 回目の修正}
  2. {2 回目の修正}
  3. {3 回目の修正}
- 残存エラー: {エラーメッセージ or 要約}
- 推測される原因: {仮説}
- ユーザーへの確認事項: {必要な判断}
```
